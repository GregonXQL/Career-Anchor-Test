## 1. 项目概述

本项目为一款微信小程序，向用户提供施恩（Edgar H. Schein）职业锚测评服务。用户凭有效邀请码作答 40 道题，系统按八种职业锚类型计分，生成包含八维雷达图、各维度分数条形图、得分最高三项职业锚详细解析的测评报告。用户可查看历史测评结果（仅存分数，报告在查看时实时渲染）。管理员可通过密码登录进入控制面板，查看所有用户的测评结果并生成/管理邀请码。

核心设计原则：

- **计分在服务端**：前端只上传原始作答，分数由后端计算并落库，防止篡改。
- **报告即渲染**：数据库只存每次测评的 8 个维度分数（及原始作答），雷达图与文字解析在查看时由前端根据分数 + 解析文案库实时生成，不存图片。
- **邀请码原子消耗**：邀请码在“提交测评”时才被消耗（而非验证时），并通过数据库原子更新防止超发。

## 3. 总体工作流

```
打开小程序
   │
   ▼
启动页 / 登录页 ──（微信静默登录，首次自动注册）
   │
   ├─────────────► 管理员入口（页面角落低调放置）
   │                    │
   │                    ▼
   │              管理员鉴权（输入管理密码）
   │                    │ 验证通过，签发管理员 Token
   │                    ▼
   │              管理员控制面板
   │                    ├─ 全部测评结果（分页、搜索、查看单条报告）
   │                    ├─ 邀请码管理（批量生成 / 列表 / 停用）
   │                    └─ 数据概览（测评总数、活跃邀请码数等）
   │
   ▼
首页（开始测试 / 查看历史结果）
   │
   ├── 开始测试
   │      │
   │      ▼
   │   邀请码输入页 ──► 后端校验（有效 / 未过期 / 未用尽）
   │      │ 校验通过（此时不消耗次数）
   │      ▼
   │   答题页（40 题，逐题作答，可回退，进度条）
   │      │ 全部作答完成
   │      ▼
   │   附加分选择页（从高分题中勾选最吻合的 3 题，各 +4 分）
   │      ▼
   │   提交（携带邀请码 + 40 条作答 + 3 个附加题号）
   │      │ 后端：再次校验并【原子消耗】邀请码 → 计分 → 落库
   │      ▼
   │   结果报告页（雷达图 + 分数 + Top3 详解）
   │
   └── 查看历史结果
          │
          ▼
       历史列表页（按时间倒序，显示日期 + Top1 锚）
          │ 点击某条
          ▼
       结果报告页（与上面同一页面，传 resultId 复用）
```

关键流程决策说明：

- **验证与消耗分离**：邀请码在进入答题前只做“校验”（不扣次数），在提交答卷时才真正“消耗”。这样用户中途退出不会浪费邀请码；代价是提交时可能出现“码刚被别人用完”的极端情况，需给出明确报错并允许换码重交（前端本地暂存作答）。
- **管理员也是微信用户**：管理员先完成微信登录，再输入密码获得管理员 Token（短有效期）。这样管理端复用同一套小程序，不需要单独 Web 后台（后续如需 Web 后台可平滑扩展，接口已按 RESTful 设计）。

## 4. 技术栈选型

### 4.1 前端：微信原生小程序（推荐）

| 项 | 选型 | 理由 |
|---|---|---|
| 框架 | 微信原生小程序（WXML/WXSS/JS，可选 TypeScript） | 只面向微信单端，原生方案文档最全、坑最少、包体最小，官方能力（canvas、隐私授权）第一时间可用。你前端经验有限，原生 + 官方文档是学习成本最低的路径 |
| UI 组件库 | Vant Weapp | 提供按钮、单元格、弹窗、进度条、折叠面板、Toast 等成熟组件，配色统一，能快速做出“像样”的界面 |
| 图表 | echarts-for-weixin（ec-canvas 组件） | ECharts 的 radar 系列可直接复刻示例报告里的八维雷达图 + 分数条形图 |
| 状态管理 | 页面级 data + 全局 `getApp().globalData` + 本地缓存 `wx.storage` | 项目规模不需要引入额外状态库 |
| 网络层 | 自封装 `request.js`（统一注入 Token、统一错误处理、401 自动跳登录） | — |

> 备选：如果未来想同一套代码发布到支付宝小程序 / H5，可改用 uni-app（Vue3）。当前需求只有微信端，不建议为未来可能性引入跨端框架的复杂度。

### 4.2 后端

| 项 | 选型 |
|---|---|
| 语言/运行时 | Java 17 |
| 框架 | Spring Boot 3.x（spring-web、spring-validation） |
| ORM | MyBatis-Plus（对你这种 Java 背景上手快，CRUD 样板代码少；也可换 Spring Data JPA，二选一即可） |
| 鉴权 | JWT（jjwt 库），无状态；用户 Token 7 天，管理员 Token 2 小时 |
| 密码哈希 | BCrypt（spring-security-crypto，只引 crypto 模块，不引整个 Security 过滤链） |
| 微信登录 | 后端调用 `https://api.weixin.qq.com/sns/jscode2session` 换取 openid（appid/secret 存服务端配置，绝不下发前端） |
| 接口文档 | springdoc-openapi（Swagger UI），开发期用，生产关闭 |
| 日志 | Logback + 按天滚动文件 |

### 4.3 数据库与部署

| 项 | 选型 |
|---|---|
| 数据库 | MySQL 8.0，utf8mb4，InnoDB |
| 连接池 | HikariCP（Spring Boot 默认） |
| 部署 | 任一云服务器（如轻量应用服务器）+ Nginx 反代 + HTTPS（小程序强制要求 request 域名为已备案的 HTTPS 域名） |
| 版本管理 | Git；数据库变更用 Flyway 管理迁移脚本（V1__init.sql 起步） |

不引入 Redis：并发规模小，邀请码原子性用 MySQL 行级更新即可；JWT 无状态无需会话存储。如后续需要“管理员强制下线/Token 吊销”，再考虑引入。

## 6. 前端页面设计与美化方案

统一视觉规范（全局 app.wxss 定义 CSS 变量）：

| 项 | 值（建议） |
|---|---|
| 主色 | 靛蓝 `#4F6BF0`（专业、测评感），渐变辅助 `#4F6BF0 → #7A5AF8` |
| 强调色 | 珊瑚橙 `#FF7A59`（用于 Top1 锚、CTA 按钮） |
| 背景 | `#F6F7FB` 浅灰底 + 白色圆角卡片（16rpx 圆角、柔和投影） |
| 字体 | 系统默认；数字用 `font-weight: 600` 突出 |
| 动效 | 页面切换默认；答题选项点击缩放反馈；报告页分数条形图从 0 增长动画（ECharts 自带） |

各页面要点：

1. **启动/登录页**：顶部品牌插画（可用免费插画库 unDraw 风格自绘）+ 一句 slogan + “微信一键进入”按钮（触发 `wx.login` 静默登录，无需用户操作授权头像昵称；昵称头像改为可选完善）。页面底部小字“管理员入口”。
2. **首页**：两张大卡片——“开始测评”（渐变主色卡，含预计用时“约 8 分钟 · 40 题”）与“历史报告”（白卡，显示已测次数角标）。底部免责声明小字：“测评结果仅供参考，不构成任何诊断或决策依据”。
3. **邀请码输入页**：居中 6 位分格输入框（类似验证码输入），自动大写、自动去空格；错误时输入框抖动 + Toast 提示具体原因（无效/过期/已用尽）。
4. **答题页**：
   - 顶部进度条（`当前题号/40`）+ Vant Progress；
   - 单题卡片式，一屏一题，题干下方为量表选项（分值按第 9 节确认后的制式渲染，选项文案如“完全不符合 … 完全符合”）；
   - 选中后自动滑入下一题（延迟 300ms），支持左滑/按钮回退修改；
   - 作答实时写入 `wx.setStorage`（key: `quiz_draft`），意外退出可恢复；
   - 第 40 题作答完成后进入**附加分选择页**：将 40 题按用户打分降序列出（默认只展示打 5–6 分的题，可展开全部），要求从中勾选与日常想法最吻合的**恰好 3 题**（每题 +4 分），选够 3 题方可继续；
   - 随后进入“提交”确认页，列出未答题目（正常流程不会有）。
5. **结果报告页**（新测评完成与历史查看共用，参数 `resultId`）：
   - 头部渐变横幅：Top1 锚名称 + 一句话定位；
   - 模块一：八维分数条形图（由高到低排序，前三名暖色、其余冷色，与示例报告一致的信息层次）；
   - 模块二：八维雷达图（ECharts radar，indicator 上限 100）；
   - 模块三：Top3 详解，每个锚一个 Vant Collapse 折叠面板，内含“特点描述 / 职场优势 / 潜在挑战 / 发展建议 / 适合职业”五个小节（内容来自解析文案库）；
   - 底部：测评时间、编号（result id 格式化）、免责声明；
   - 右上角“保存长图”按钮（二期功能，见第 10.3 节）。
6. **历史列表页**：时间倒序卡片流，每条显示日期、Top1 锚徽章（每种锚固定一个颜色）、Top3 缩略（如“挑战 · 创造 · 服务”）；点击进入报告页。空态插画 + “去测一次”按钮。
7. **管理员密码页**：单密码输入 + 提交；连续失败 5 次锁定 10 分钟（后端控制）。
8. **管理员控制面板**：底部 Tab 三栏——
   - 概览：测评总数、今日测评数、用户数、活跃邀请码数（4 个统计卡）；
   - 结果：全部测评列表（昵称/openid 尾号、时间、Top1），支持按昵称搜索、时间筛选，点击查看完整报告；
   - 邀请码：生成表单（数量 1–100、有效期、备注；每码可用次数固定为 1，即**一码一次**）→ 生成后列表展示并支持一键复制全部；列表含状态（有效/已使用/已过期/已停用）、操作（停用）。`max_uses` 字段保留在库表与接口中（固定填 1），将来若需要“一码多人”只需放开前端表单。

## 7. 数据库设计

### 7.1 ER 概览

```
users 1 ──── n test_results n ──── 1 invite_codes
users 1 ──── n invite_code_usages n ──── 1 invite_codes
questions（题库，独立）
anchor_profiles（解析文案库，独立，主键=锚编码）
admin_accounts（管理员账号，独立）
```

### 7.2 建表 SQL（Flyway V1__init.sql）

```sql
CREATE DATABASE IF NOT EXISTS career_anchor
  DEFAULT CHARACTER SET utf8mb4 COLLATE utf8mb4_unicode_ci;
USE career_anchor;

-- 用户表
CREATE TABLE users (
  id            BIGINT UNSIGNED AUTO_INCREMENT PRIMARY KEY,
  openid        VARCHAR(64)  NOT NULL COMMENT '微信 openid',
  nickname      VARCHAR(64)  DEFAULT NULL,
  avatar_url    VARCHAR(512) DEFAULT NULL,
  created_at    DATETIME     NOT NULL DEFAULT CURRENT_TIMESTAMP,
  last_login_at DATETIME     DEFAULT NULL,
  UNIQUE KEY uk_openid (openid)
) ENGINE=InnoDB COMMENT='微信用户';

-- 管理员账号表（密码 BCrypt 哈希，初始账号由运维 SQL 插入）
CREATE TABLE admin_accounts (
  id            BIGINT UNSIGNED AUTO_INCREMENT PRIMARY KEY,
  username      VARCHAR(32)  NOT NULL DEFAULT 'admin',
  password_hash VARCHAR(100) NOT NULL,
  created_at    DATETIME     NOT NULL DEFAULT CURRENT_TIMESTAMP,
  UNIQUE KEY uk_username (username)
) ENGINE=InnoDB COMMENT='管理员';

-- 邀请码表
CREATE TABLE invite_codes (
  id          BIGINT UNSIGNED AUTO_INCREMENT PRIMARY KEY,
  code        VARCHAR(16)  NOT NULL COMMENT '码值，如 8 位大写字母数字',
  max_uses    INT UNSIGNED NOT NULL DEFAULT 1 COMMENT '可用总次数',
  used_count  INT UNSIGNED NOT NULL DEFAULT 0 COMMENT '已用次数',
  expires_at  DATETIME     DEFAULT NULL COMMENT 'NULL=永不过期',
  status      ENUM('ACTIVE','DISABLED') NOT NULL DEFAULT 'ACTIVE',
  remark      VARCHAR(128) DEFAULT NULL,
  created_by  BIGINT UNSIGNED DEFAULT NULL COMMENT '生成者 admin_accounts.id',
  created_at  DATETIME     NOT NULL DEFAULT CURRENT_TIMESTAMP,
  UNIQUE KEY uk_code (code),
  KEY idx_status (status)
) ENGINE=InnoDB COMMENT='邀请码';

-- 邀请码使用记录（审计用；一码一次策略下每码至多一条记录）
CREATE TABLE invite_code_usages (
  id             BIGINT UNSIGNED AUTO_INCREMENT PRIMARY KEY,
  invite_code_id BIGINT UNSIGNED NOT NULL,
  user_id        BIGINT UNSIGNED NOT NULL,
  used_at        DATETIME NOT NULL DEFAULT CURRENT_TIMESTAMP,
  KEY idx_code (invite_code_id),
  KEY idx_user (user_id)
) ENGINE=InnoDB COMMENT='邀请码使用记录';

-- 题库表（40 题种子数据，见附录 A）
CREATE TABLE questions (
  id          INT UNSIGNED PRIMARY KEY COMMENT '即题号 1-40',
  content     VARCHAR(512) NOT NULL COMMENT '题干',
  anchor_code VARCHAR(20)  NOT NULL COMMENT '所属职业锚编码',
  enabled     TINYINT(1)   NOT NULL DEFAULT 1,
  KEY idx_anchor (anchor_code)
) ENGINE=InnoDB COMMENT='题库';

-- 解析文案库（每锚一行，JSON 存结构化文案，管理端后续可做编辑功能）
CREATE TABLE anchor_profiles (
  anchor_code VARCHAR(20) PRIMARY KEY,
  name_cn     VARCHAR(32)  NOT NULL,
  name_en     VARCHAR(64)  NOT NULL,
  tagline     VARCHAR(128) NOT NULL COMMENT '一句话定位，用于报告头部与八型简介',
  traits      JSON NOT NULL COMMENT '特点描述 [{"title":"...","desc":"..."}]',
  strengths   JSON NOT NULL COMMENT '职场优势，同上结构',
  risks       JSON NOT NULL COMMENT '潜在挑战，同上结构',
  advices     JSON NOT NULL COMMENT '发展建议，同上结构',
  careers     JSON NOT NULL COMMENT '适合职业 [{"title":"项目经理","desc":"..."}]'
) ENGINE=InnoDB COMMENT='职业锚解析文案';

-- 测评结果表（核心：只存分数与原始作答，报告实时渲染）
CREATE TABLE test_results (
  id                BIGINT UNSIGNED AUTO_INCREMENT PRIMARY KEY,
  user_id           BIGINT UNSIGNED NOT NULL,
  invite_code_id    BIGINT UNSIGNED NOT NULL,
  scale_max         TINYINT UNSIGNED NOT NULL COMMENT '本次测评量表上限（当前为 6），保证历史可复算',
  answers           JSON NOT NULL COMMENT '原始作答 {"items":[{"q":1,"v":4},...共40条],"boosted":[9,23,31]}',
  score_technical   TINYINT UNSIGNED NOT NULL COMMENT '各维度原始分=5题之和+附加分，范围5-42',
  score_managerial  TINYINT UNSIGNED NOT NULL,
  score_autonomy    TINYINT UNSIGNED NOT NULL,
  score_security    TINYINT UNSIGNED NOT NULL,
  score_creativity  TINYINT UNSIGNED NOT NULL,
  score_service     TINYINT UNSIGNED NOT NULL,
  score_challenge   TINYINT UNSIGNED NOT NULL,
  score_lifestyle   TINYINT UNSIGNED NOT NULL,
  top1              VARCHAR(20) NOT NULL,
  top2              VARCHAR(20) NOT NULL,
  top3              VARCHAR(20) NOT NULL,
  created_at        DATETIME NOT NULL DEFAULT CURRENT_TIMESTAMP,
  KEY idx_user_time (user_id, created_at DESC),
  KEY idx_time (created_at DESC),
  CONSTRAINT fk_result_user   FOREIGN KEY (user_id) REFERENCES users(id),
  CONSTRAINT fk_result_invite FOREIGN KEY (invite_code_id) REFERENCES invite_codes(id)
) ENGINE=InnoDB COMMENT='测评结果';
```

设计说明：

- `test_results` 存**原始分**（5 题之和 + 落在该维度的附加分），百分制换算在读取时进行（见 9.3），这样若未来调整换算公式，历史数据不受影响。
- 同时冗余存 `answers` JSON：满足“只存分数”的最小需求之上，多花极少空间换取可审计、可复算（比如以后想改计分规则重跑历史）。若你坚持只存分数，删掉该列即可，其余设计不变。
- `top1/2/3` 冗余落库：历史列表页直接查表渲染徽章，不必每次解 JSON 排序。
- 外键在低并发场景下保留以保数据一致；若后续压测有性能顾虑可移除、改由应用层保证。

### 7.3 种子数据

- `questions`：40 行，题干已按原文全部填入（见附录 A），可直接执行。
- `anchor_profiles`：8 行，原创解析文案已撰写完毕（见附录 C），可直接执行；后续可在库中随时微调措辞。
- `admin_accounts`：1 行，密码用 BCrypt 生成哈希后插入。

### 8.4 邀请码生成与消耗

生成：8 位、去易混字符的字母数字（字符集 `23456789ABCDEFGHJKMNPQRSTUVWXYZ`，去掉 0/O/1/I/L），随机生成后查重重试。

消耗（提交答卷事务内）：

```sql
UPDATE invite_codes
SET used_count = used_count + 1
WHERE code = ?
  AND status = 'ACTIVE'
  AND (expires_at IS NULL OR expires_at > NOW())
  AND used_count < max_uses;
```

受影响行数为 1 才继续；为 0 则按具体原因回查并返回 41002/41003/41004。单条 UPDATE 自带原子性，无需悲观锁。

### 8.5 计分服务（核心逻辑）

```java
public ScoreResult score(List<Answer> answers, List<Integer> boosted, int scaleMax) {
    // 1. 校验：40 条、题号 1..40 无重复、每个 v ∈ [1, scaleMax(=6)]；
    //    boosted 恰好 3 个、互不相同、均为合法题号
    // 2. 附加分：boosted 中每题得分 +4（例：原 5 分 → 9 分）
    // 3. 按题库 anchor_code 分组求和 → 8 个原始分（每锚 5 题，范围 5–42）
    // 4. 平均分 = 原始分 / 5（原文判定口径，最高者即职业锚）
    // 5. 百分制展示分 = min(100, round(原始分 / (5 * scaleMax) * 100))
    // 6. 排序取 Top3：按原始分降序；同分按固定优先级
    //    （题组序：TECHNICAL > MANAGERIAL > AUTONOMY > SECURITY >
    //      CREATIVITY > SERVICE > CHALLENGE > LIFESTYLE）保证结果确定性
}
```

计分为纯函数，必须配 JUnit 单元测试：满分卷、最低分卷、同分并列、非法输入各一组用例。

## 9. 测试题目与计分方式

### 9.1 题目来源与结构（已确认定稿）

采用知乎文章《施恩职业锚测评量表》（zhuanlan.zhihu.com/p/512735297）中的 **40 题版本**，题干全文已录入附录 A。每题为一条职业描述，被测者按符合程度打 **1–6 分**：

| 分值 | 含义 |
|---|---|
| 1 | 完全不符合 |
| 2 | 偶尔这么想 |
| 3 | 有时这么想 |
| 4 | 经常这么想 |
| 5 | 频繁这么想 |
| 6 | 完全符合（日常想法） |

原文的作答引导语提示“除非非常明确，否则不需要做出 1 或 6 的极端选择”，该提示在答题页首屏展示一次。

40 题作答完成后进入**附加分环节**：从自己给分较高的题目中，挑出与日常想法最吻合的 **3 题，各额外 +4 分**（例：原 5 分 → 调整后 9 分）。

### 9.2 题号 → 职业锚映射

40 题按 8 型循环排列，每型 5 题（已与原文全部 40 题逐一核对，映射一致）：

| 职业锚 | 题号 |
|---|---|
| 技术/职能型 TECHNICAL | 1, 9, 17, 25, 33 |
| 管理型 MANAGERIAL | 2, 10, 18, 26, 34 |
| 自主/独立型 AUTONOMY | 3, 11, 19, 27, 35 |
| 安全/稳定型 SECURITY | 4, 12, 20, 28, 36 |
| 创造/创业型 CREATIVITY | 5, 13, 21, 29, 37 |
| 服务型 SERVICE | 6, 14, 22, 30, 38 |
| 挑战型 CHALLENGE | 7, 15, 23, 31, 39 |
| 生活型 LIFESTYLE | 8, 16, 24, 32, 40 |

映射示例：第 17 题（资深专家 vs 总经理）→ 技术/职能型；第 34 题（不愿离开全面管理）→ 管理型；第 40 题（最小化个人与家庭冲突）→ 生活型。

### 9.3 计分公式（已按原文确认：1–6 制 + 3 题附加分）

1. **附加分**：用户选定的 3 题各 +4 分；
2. **单锚原始分** = 该锚 5 题得分之和（含附加分），范围 [5, 42]（30 为无附加分满分；附加分最多可为同一锚贡献 +12）；
3. **单锚平均分** = 原始分 ÷ 5，范围 [1.0, 8.4]——这是原文的判定口径，**最高平均分即为该用户的职业锚**（由于都除以 5，按平均分排序与按原始分排序等价；报告可在分数条旁以小字展示，如“均分 6.4”）；
4. **百分制展示分** = `min(100, round(原始分 / 30 × 100))`，用于雷达图与条形图的 0–100 刻度。附加分可能使原始分超过 30，展示时封顶 100；**排序始终按原始分**（而非封顶后的展示分），保证判定与展示顺序一致；
5. **Top3** = 原始分降序前三，同分按 9.2 表格顺序（题组序）裁定，保证结果确定性。

`scaleMax=6`、`boostCount=3`、`boostValue=4` 与量表选项文案均为后端配置（`application.yml` → 经 `/questions` 接口下发），前端不硬编码；`test_results.scale_max` 落库保证规则日后调整时，历史记录仍按当时制式正确展示。

### 9.4 作答体验规则

- 必答全部 40 题，且附加分环节必须恰好勾选 3 题，方可提交；
- 附加分选择页按用户打分降序列题（默认展示打 5–6 分的题，可展开全部），引导用户从高分题中挑选；
- 允许回退修改（若回退修改了已勾选附加题的分数，勾选保持不变，按新分数 +4 计算）；
- 前端本地草稿：每次选择即存 `wx.storage`（含附加题勾选），重新进入答题页时提示“继续上次作答 / 重新开始”；
- 提交成功后清除草稿。

## 11. 安全设计

- appid/secret、JWT 密钥、数据库口令全部走服务器环境变量或 `application-prod.yml`（不入 Git）。
- 管理员密码仅存 BCrypt 哈希；登录失败计数 + 锁定；管理员 Token 短有效期（2h）。
- 所有写接口做参数校验（spring-validation 注解 + 全局异常处理器统一返回 40001）。
- 提交答卷做频控：同一用户 60 秒内最多提交 1 次（内存计数即可），防止脚本刷。
- `/results/{id}` 严格校验资源归属，防止水平越权（横向遍历他人 id）。
- Nginx 层限速（如 `limit_req` 10r/s per IP）；仅开放 443。
- 日志不打印 openid 全文与邀请码明文（打码处理）。

## 12. 合规与上架注意事项

1. **服务类目**：小程序上架需选择与测评服务匹配的服务类目；心理/测评相关类目在微信平台可能要求主体资质（个人主体可选类目更受限）。开发前先在小程序后台确认你的主体类型能过审的类目，这直接影响能否上线，建议第一时间验证。
2. **隐私合规**：需在小程序后台配置《用户隐私保护指引》，声明收集 openid、昵称、头像及测评作答数据的用途；首次启动弹隐私授权弹窗（基础库已内置 `wx.requirePrivacyAuthorize` 流程）。
3. **免责声明**：参照示例报告，在首页与报告页固定展示“测评结果仅供参考，不构成任何诊断”类声明。
4. **内容版权**：题目量表源自施恩教授的问卷（网络流传中文版）；报告解析文案不可复制示例图中拾棠平台的文案，本文档附录 C 已提供一套可直接使用的原创文案。
5. **域名**：request 合法域名必须 HTTPS + ICP 备案，需提前办理（备案周期 1–2 周，是排期上的长杆项）。
